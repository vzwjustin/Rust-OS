# RustOS Codebase Audit — 2026-07-02

Whole-codebase audit (correctness, security, build health, architecture) run via 6 parallel review agents over: core kernel, filesystem layer, drivers/hardware, network stack, mutter/desktop port, and the linux_compat/syscall security boundary. Build compiles clean (0 errors, 6840 warnings) at the time of this audit.

All findings below were spot-verified against source before being recorded here; two "unresolved merge conflict" claims and one "no fork isolation" over-claim from the raw agent output were corrected or downgraded during verification.

## Cross-cutting theme: duplicate parallel subsystems, only one path live

The rapid multi-agent build-out left 2-3 parallel implementations of several core subsystems, with only one path actually wired into the live runtime. Landmines in the dead paths will detonate if anyone wires them in without re-auditing.

- [ ] **Syscall dispatch x3** — live path is `syscall_handler.rs -> linux_integration::route_syscall -> linux_compat::*`. `src/syscall/mod.rs` and `src/process/syscalls.rs::SyscallDispatcher` are dead code (the latter is constructed in `ProcessManager::new()` but `.dispatch()` has zero call sites — verified). Decide which is canonical and delete the others.
- [ ] **VFS x2** — `crate::vfs` (live, used by linux_compat) vs. `crate::fs` (30-module FileSystem-trait layer, reachable only through the dead `process/syscalls.rs` path).
- [ ] **`wait4` x2** — `process_ops::wait4` (live) vs. `syscall/mod.rs::sys_wait4` (dead, has a real wstatus-encoding bug at line 1478).
- [ ] **Mutter/desktop x2** — `src/mutter_port/` (534 files, ~89k LOC) is never declared as a module anywhere (no `mod mutter_port;` exists) — 100% dead code, contradicting its own `TODO.md` which claims it's wired into the build. Live desktop path is `src/mutter.rs` + `src/desktop/*` + `src/gnome.rs`. Either wire it in (after fixing the TODO.md mismatch) or delete it.

## Live, high-severity findings

- [ ] **`hwmon` reads temperature/voltage MSRs with no CPUID gate** — `src/drivers/hwmon/mod.rs:106,124,169`. GPFs on any non-Intel/hypervisor CPU. Exact same bug class just fixed in `thermal/mod.rs`; the fix never propagated to this sibling module. `hwmon::init()` runs unconditionally at boot. **Highest-confidence, cheapest fix.**
- [ ] **TCP has no admission control** — `src/net/tcp.rs`. Unbounded half-open connection table (SYN flood -> OOM, no SYN cookies), unbounded out-of-order segment buffer, `recv_buffer` grows without limit while advertised window never shrinks. Packet parsing itself is solid (bounds-checked). Fix: global connection cap + half-open timeout/SYN-cookies, bound ooo_segments, make recv_window track free buffer space.
- [ ] **Fork's copy-on-write fails silently** — `src/process/integration.rs:630-680`. CoW is attempted for code/data/heap via `clone_page_entries_cow`, but the Result is discarded (`let _ = ...`) assuming "child gets fresh mappings on exec." A fork-without-exec child could end up with unmapped heap or unprotected shared pages if the call fails.
- [ ] **PS/2 mouse ISR vs. non-ISR code share a blocking spinlock** — `src/drivers/ps2_mouse.rs:382` (`MOUSE_STATE`). Same deadlock class as the timer/TIMER_MANAGER bug fixed earlier this session. Boot-time init is safe (IRQs masked); runtime access with IRQs enabled is not.
- [ ] **Ignored `Result`s on subsystem-critical ops** — `src/drivers/power/mod.rs:136-168` (suspend/freeze/poweroff/resume — a failed suspend can be recorded as success, then resume runs against hardware that never suspended). `src/process/fork.rs:334` is lower priority — traced to a vestigial always-`Ok` inner call.
- [ ] **`timer_settime` validation is dead + integer overflow** — `src/linux_compat/time_ops.rs:392-419`. Negative-value checks are no-ops on unsigned fields; `sec * 1_000_000_000` has no overflow guard (debug: panic/DoS, release: silent wrap).
- [ ] **Seccomp filters installed but never enforced** — re-confirmed from the 2026-06-30 audit. `dispatch_syscall`'s only gate is privilege-ring validation; seccomp state is never consulted on the live path.
- [ ] **No signal delivery on return to userspace (PLAUSIBLE, needs confirmation)** — `kill()` only enqueues into `pending_signals`; no `do_signal`-style deliver-on-return hook found in the live path. Registered handlers may never fire.
- [ ] **`do_exit` halts the CPU without switching away** — `src/process/exit.rs:333-338`. `pm.schedule()` only selects the next PID, then falls through to `cli; hlt`. If this is the real exit path, exiting wedges the CPU.

## Dead-code landmines (leave alone, but fix before ever wiring in)

- [ ] `sys_wait4` in `src/syscall/mod.rs:1478` — wrong wstatus encoding, breaks `WIFEXITED`/`WEXITSTATUS`.
- [ ] `sys_brk` in `src/process/syscalls.rs:1166` — allocates memory but discards the returned VA, never actually maps it.
- [ ] `send_signal` in `src/process/ipc.rs:~665` — drops blocked signals instead of queueing them (violates POSIX, would let SIGKILL/SIGSTOP be masked).
- [ ] `fs::VfsFile::write` path — writes go to an in-RAM `Vec` only, never flushed to the mounted ext4; `VFS::open`/`stat` eagerly buffer the whole file (OOM on a crafted large-size inode).
- [ ] Entire `src/rcu/` subsystem — fully implemented, zero callers, protects nothing.
- [ ] `src/smp.rs::start_ap` — zero callers, secondary CPUs never start, kernel silently runs single-core.

## Build health

6840 warnings at audit time; ~98.6% is expected `dead_code` from scaffolding (low priority). Actionable tail: 8 ignored `Result`s (all subsystem-critical, listed above), 5 dead comparisons (time_ops/ioctl_ops), 76 auto-fixable via `cargo fix`.

## Suggested order of work

1. Gate `hwmon`'s MSR reads the same way `thermal` was fixed — smallest, most certain crash fix.
2. Add a global cap + SYN-cookie fallback to TCP connection admission.
3. Decide which syscall dispatcher / VFS stack is canonical and delete the others — the duplication is the root cause feeding most of the "latent" findings above.
4. Either wire `src/mutter_port/` in (after fixing its `TODO.md` mismatch) or delete it.
